COMPONENT 01

TEACH YOUR ASSISTANT YOUR STACK

Skills are knowledge modules that extend your agent's expertise. Think of them as persistent context that teaches the AI your framework conventions, design patterns, and best practices. Instead of explaining Django DRF patterns in every conversation, load the django-expert skill once and it remembers.

🐍
django-expert

Django best practices, DRF patterns, ORM optimization, security defaults.

⚛️
react-best-practices

Modern React patterns, hooks, performance optimization, accessibility.

🔍
systematic-debugging

Structured debugging methodology: hypothesize, test, isolate, fix.

🎮
godot-gdscript

Godot 4.x patterns, GDScript idioms, scene architecture, signal usage.

🎨
frontend-design

Design systems, component libraries, responsive patterns, accessibility.

🔒
security-audit

OWASP Top 10, auth patterns, input validation, secure defaults.

COMPONENT 02

SPECIALISTS, NOT GENERALISTS

Custom Agents are specialist personas with specific roles, tools, and expertise. Instead of one assistant trying to do everything, you build a team: a code reviewer who's thorough and critical, an implementer who follows specs exactly, a researcher who explores thoroughly before deciding.

👁️
code-review

Finds bugs, security issues, and logic errors. Never comments on style.

📋
spec-writer

Transforms ideas into detailed, unambiguous specifications.

⚙️
implementer

Follows specs exactly. Tests every acceptance criterion.

🎨
frontend-impl

Turns designs into pixel-perfect, accessible components.

🔬
ux-research

Creates personas, journey maps, and competitive analysis.

🎯
ui-lifecycle-master

Orchestrates the complete UI/UX pipeline: research → IA → wireframes → UX → UI → implementation → testing. Coordinates 7 specialized agents with automatic quality gates between phases.

agent-definition.yaml 
name: code-review
type: specialist
model: claude-opus-4.6

personality:
  tone: direct, thorough, zero-tolerance for bugs
  focus: logic errors, security, potential crashes
  ignore: style, formatting, trivial matters

tools:
  - grep # search codebase
  - view # read files
  - powershell # run tests

skills:
  - security-audit
  - systematic-debugging
  - rust-ownership # if reviewing Rust

rules:
  - "Never comment on code style or formatting"
  - "Only surface issues that genuinely matter"
  - "Include fix suggestions, not just complaints"
COMPONENT 03

ORCHESTRATE YOUR AGENT TEAM

Workflows chain agents together into automated pipelines. A user story becomes a spec (via spec-writer), the spec becomes implementation (via implementer), and the implementation gets reviewed (via reviewer). Each agent hands off to the next, with quality gates to catch issues early.

workflow-sdd-pipeline.yaml 
name: spec-driven-development
description: "Idea → Spec → Implementation → Review"

steps:
  - agent: spec-writer
    input: ${user_idea}
    output: spec.md
    quality_gate:
      - check: "Contains user stories"
      - check: "Has acceptance criteria"
      - check: "Defines edge cases"

  - agent: implementer
    input: spec.md
    output: implementation/
    skills:
      - react-best-practices
      - typescript-patterns
    quality_gate:
      - check: "All tests pass"
      - check: "Code builds without errors"

  - agent: reviewer
    input: [spec.md, implementation/]
    output: compliance-report.html
    quality_gate:
      - check: "Compliance score >= 90%"

on_failure:
  - notify: slack
  - rollback: true

COMPONENT 05

YOUR CODEBASE, YOUR RULES

Rules are persistent constraints that shape agent behavior. They're like .eslintrc for AI: enforce code style, architectural patterns, security policies, and team conventions. Rules persist across sessions—define once, enforced forever.

WITHOUT RULES
// Agent suggests generic patterns
function fetchUser(id) {
  console.log('Fetching user:', id);
  return fetch(`/api/users/${id}`)
    .then(res => res.json());
}

// Uses whatever naming it feels like
const userData = await fetchUser(123);

// No consistency across suggestions
WITH RULES
// Follows your team's patterns exactly
async function fetchUserById(userId: string): Promise<User> {
  return apiClient.get(`/users/${userId}`);
}

// Matches your naming conventions
const user = await fetchUserById('123');

// Enforces: async/await, TypeScript, no console.log
.antigravity/rules.md 
# Project Rules — Enforced Across All Agents

## Code Style
- Use async/await, never .then() chains
- TypeScript for all new code
- No console.log in production code (use logger.info)
- Destructure props in function signatures

## Architecture
- API calls must go through apiClient wrapper
- React components: presentational vs. container pattern
- State management: Zustand only, no Redux
- File naming: kebab-case for files, PascalCase for components

## Security
- Never concatenate user input into SQL queries
- All API endpoints require authentication by default
- Sanitize user input before rendering (XSS prevention)
- Use environment variables for secrets, never hardcode

## Testing
- Every new feature requires tests
- Test file naming: ComponentName.test.tsx
- Mock external API calls in tests
- Aim for 80% coverage on new code

# These rules persist across all agent sessions
# Agents will follow them automatically
